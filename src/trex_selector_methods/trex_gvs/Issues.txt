Issue 1: The `finalizeSetup()` method.
The C++ code builds the empricial covariance as Sigma_m as (1/(n−1)) X^T * Xm.
X_ is already L2-normalized from the base class, and it even notes that each column has norm 1
before multiplying by 1/(n-1).
In the reference R code we used z-score standardization originally, but now we want to unit-l2-norm
standardize the data at first.
But we must be sure, that the factor 1/(n-1) does not makes the dummies to weak.
In the R code also the z-score standardization was used, but we want to stick with the new data
normalization in the base trex class: unit-norm X and centering (as it is already done).

What should be updated is the conceptual reference: the old R code based on z-scores is no longer
the gold standard for this block unless you rewrite the R preprocessing to the same unit-norm
convention first.
If you want an extra safeguard, the right diagnostic is to simulate dummy layers and compare their
empirical column norms and within-cluster correlations against those of normalized X_; that would
test the scale directly instead of relying on the old z-score intuition.


Issue 2: The `buildIENAugmentation()` method.
The R IEN augmentation multiplies the bottom block by sqrt(lambda_2_lars) through
X_aug <- sqrt(lambda_2_lars) * rbind((1/sqrt(lambda_2_lars))*X_tilde, bottom_block), but the C++
buildIENAugmentation() stores plain ien_bottom_block_p_ into X_aug_ with no lambda2_ scaling.
Since the IEN branch then uses TLASSO without a separate lambda2 argument, the penalty strength
is effectively missing from the augmented system.
